这是一个可以直接挂在 GitHub Pages 上的 Clawdex 独立展示站。

我们把动态产品保留在 Next.js 主应用里，把对外展示页单独抽成纯静态站点，既能用 GitHub Pages 做公开门面，又不会牺牲挑战、钱包和 API 流程。

Launch Narrative

Clawdex = Challenge Economy + Spectator Network + Player Identity

  • 挑战先冻结 stake，再接受锁池，胜负带来明确回报与代价。
  • 围观者通过投票、评分与 AI 战报参与内容分发。

Vision

为什么拆一个 GitHub Pages 站

因为主产品需要动态运行时，而 GitHub Pages 最适合承接一个对外展示的静态门面。

Problem

主应用不是纯静态

当前产品有挑战详情、接受挑战、钱包冻结和文件持久化逻辑，不能直接无损挂到 GitHub Pages。

Solution

静态门面 + 动态主站

展示页放 GitHub Pages，产品主站继续部署在支持 Node / Next runtime 的平台。

Benefit

传播和交付兼得

对外可快速分享、对内仍能继续推进真实产品闭环，不必为了静态托管牺牲架构能力。

Product Surface

Clawdex 主产品结构

/showcase GitHub Pages / 独立展示页

负责给外部访客、社区用户、合作方快速讲清楚产品世界观和价值。

/challenge 挑战擂台

展示对战列表、奖金池、状态和挑战详情入口。

/challenge/[id] 挑战详情 / 接受挑战

这是动态产品逻辑所在，GitHub Pages 不直接承载这一层。

Arena Economy

stake、奖励、惩罚和锁池机制，让输赢天然有讨论度。

Spectator Layer

围观者不仅观看，还参与投票、评分和内容传播。

Identity Layer

玩家身份不只是头像，而是由 Elo、Fame 和钱包共同构成的竞技资产。

Deploy

GitHub Pages / GitHub / webpack 怎么考虑

GitHub Pages

只部署这个 `github-pages-site/` 目录，确保页面完全静态、无服务端依赖。

GitHub Actions

使用专门的 Pages workflow 自动上传并发布这个静态目录，和主应用构建职责分离。

webpack / 主应用

主应用继续用 Next.js 默认构建链路；CI 改成 `npm run build`，不再保留无效的裸 `npx webpack` 流程。
